내 판단은 이렇습니다.

**이전 답변의 “ACTOR_TYPE 비활성화해도 멀쩡히 도는 건 정상 흐름”이라는 말은 반만 맞습니다.**

정확히 말하면:

> **현재 구현 기준으로는 정상 동작처럼 보이는 게 맞지만, 공통코드 관리 기능의 설계 관점에서는 위험한 상태입니다.**

즉, **버그라기보다는 정책이 명확히 정의되지 않은 설계 문제**에 가깝습니다.

---

## 1. ACTOR_TYPE은 일반 공통코드로 보면 안 됨

`ACTOR_TYPE`은 단순 드롭다운 값이 아닙니다.

예를 들면:

```text
OWNER
BROKER
OFFICER
TRANSPORT_MANAGER
WAREHOUSE_MANAGER
SYSTEM_ADMIN
```

이 값들은 단순 표시용 코드가 아니라 로그인, 권한, 라우팅, 회원 구분, 메뉴 분기와 연결됩니다.

그래서 `ACTOR_TYPE`을 관리자가 화면에서 비활성화할 수 있게 두는 건 위험합니다.

왜냐하면 관리자는 당연히 이렇게 생각합니다.

> “ACTOR_TYPE을 비활성화했으니 해당 액터 유형은 시스템에서 더 이상 안 쓰이겠지?”

그런데 실제로는 자바, Spring Security, React, Mapper 쪽에서 `"OWNER"`, `"BROKER"` 같은 문자열을 직접 쓰고 있으면 공통코드를 꺼도 시스템은 그대로 돌아갑니다.

이건 사용자 입장에서 **관리 화면이 거짓말하는 상태**가 됩니다.

---

## 2. 공통코드는 최소 3종류로 나눠야 함

지금처럼 모든 코드를 똑같이 `USE_YN`으로만 관리하면 애매해집니다.

나는 이렇게 나누는 게 맞다고 봅니다.

| 구분        | 예시                                    |    비활성화 가능 여부 | 비활성화 의미                      |
| --------- | ------------------------------------- | ------------: | ---------------------------- |
| 시스템 핵심 코드 | `ACTOR_TYPE`, `ROLE_CD`, 주요 업무 도메인 코드 |      원칙적으로 불가 | 시스템 구조에 직접 연결되므로 관리자가 끄면 안 됨 |
| 업무 선택 코드  | 문의유형, 자료실 공개대상, FAQ 카테고리, 운송수단 등      |            가능 | 신규 등록/수정 시 선택 불가             |
| 표시/라벨 코드  | 상태명 변환, 과거 데이터 라벨 표시용 코드              | 가능하더라도 조회는 유지 | 기존 데이터 이름 표시용으로는 계속 보여야 함    |

핵심은 이겁니다.

> **비활성화 = 기존 데이터 삭제/무효화가 아니라, 신규 사용을 막는 것**이어야 합니다.

---

## 3. 그룹 비활성화와 코드 비활성화 의미도 분리해야 함

### 공통코드 비활성화

예를 들어 `TRANSPORT_TYPE` 안에 이런 코드가 있다고 하면:

```text
SEA  해상
AIR  항공
LAND 육상
```

`AIR`를 비활성화하면 의미는 이래야 합니다.

```text
기존 데이터에 AIR가 들어간 것은 그대로 조회 가능
신규 등록/수정 화면에서는 AIR 선택 불가
기존 AIR 데이터는 "항공(비활성)" 정도로 표시 가능
```

즉, 기존 데이터를 깨면 안 됩니다.

---

### 공통코드 그룹 비활성화

예를 들어 `TRANSPORT_TYPE` 그룹 자체를 비활성화하면 의미는 이래야 합니다.

```text
이 그룹 전체를 더 이상 신규 입력용 코드셋으로 쓰지 않음
해당 그룹의 코드들은 드롭다운에서 전부 제외
하지만 기존 데이터 라벨 조회는 유지
```

단, `ACTOR_TYPE` 같은 시스템 핵심 그룹은 비활성화 자체를 막아야 합니다.

---

## 4. ACTOR_TYPE은 어떻게 처리하는 게 맞나

나는 `ACTOR_TYPE`은 **비활성화 버튼을 막는 게 정답**이라고 봅니다.

관리 화면에서 이렇게 처리하는 게 좋습니다.

```text
ACTOR_TYPE
ROLE_CD
SYSTEM_ADMIN 관련 권한 코드
수입/수출 업무 핵심 도메인 코드
```

이런 그룹은 “시스템 코드”로 분류하고:

```text
수정 가능: 코드명, 설명, 정렬순서 정도
수정 불가: 코드값, 사용여부, 삭제
```

로 제한하는 게 맞습니다.

즉, `ACTOR_TYPE`의 `USE_YN`을 바꾸는 기능은 제공하지 않는 게 맞습니다.

만약 꼭 화면에 보여야 한다면:

```text
시스템 핵심 코드입니다. 비활성화할 수 없습니다.
```

라고 표시해야 합니다.

---

## 5. 현재 상태에서 “비활성화해도 문제없이 돌아감”은 문제인가?

내 기준으로는 **문제 맞습니다.**

단, “시스템 장애” 문제는 아닙니다.

문제의 종류가 다릅니다.

```text
장애 문제: 아님
설계 정책 문제: 맞음
관리자 UX 문제: 맞음
운영 리스크 문제: 맞음
```

왜냐하면 관리자가 시스템 설정을 바꿨다고 생각했는데 실제 동작에는 영향이 없기 때문입니다.

이건 나중에 운영자가 이렇게 말할 수 있습니다.

> “비활성화했는데 왜 아직도 로그인돼요?”
> “비활성화했는데 왜 회원관리에서 계속 나와요?”
> “공통코드 관리가 실제로 적용되는 게 맞나요?”

이 질문이 나오는 순간 관리 기능의 신뢰도가 떨어집니다.

---

## 6. 제일 현실적인 개선안

지금 프로젝트 단계에서는 과하게 리팩토링하지 말고, 이렇게 가는 게 좋습니다.

### 1단계: 시스템 코드 보호

`COM_CODE_GRP`에 컬럼 하나 추가하는 방식이 가장 깔끔합니다.

```sql
CCG_SYSTEM_YN CHAR(1) DEFAULT 'N'
```

또는 이미 컬럼 추가가 부담되면 서버 코드에서 보호 그룹 목록을 상수로 둬도 됩니다.

```java
ACTOR_TYPE
ROLE_CD
USER_TYPE
```

이런 그룹은 비활성화 요청이 오면 서버에서 막습니다.

---

### 2단계: 드롭다운 조회는 `USE_YN = 'Y'`만

신규 등록/수정 화면의 선택 목록은 반드시 활성 코드만 가져와야 합니다.

```sql
WHERE ccg.ccg_use_yn = 'Y'
  AND cc.cc_use_yn = 'Y'
```

이건 맞습니다.

---

### 3단계: 기존 데이터 라벨 조회는 비활성 코드도 보여줌

예를 들어 기존 회원, 기존 신고서, 기존 FAQ에 이미 어떤 코드값이 저장되어 있다면, 그 코드가 비활성화되었다고 이름이 사라지면 안 됩니다.

라벨 변환 조인은 이렇게 가는 게 맞습니다.

```sql
LEFT JOIN com_code cc
       ON cc.cc_grp_cd = '...'
      AND cc.cc_cd = 대상컬럼
```

여기서는 `cc_use_yn = 'Y'`를 무조건 걸면 안 됩니다.

대신 화면 표시를 이렇게 할 수 있습니다.

```text
항공
항공(비활성)
```

---

## 7. 결론

내 결론은 명확합니다.

**ACTOR_TYPE이 비활성화돼도 시스템이 계속 돌아가는 것 자체는 현재 구조상 당연합니다. 하지만 그걸 “정상적인 공통코드 관리”라고 보면 안 됩니다.**

정리하면:

```text
ACTOR_TYPE = 공통코드 테이블에서 관리할 수는 있지만, 일반 코드처럼 비활성화하면 안 되는 시스템 코드
```

따라서 공통코드 관리 정책은 이렇게 잡는 게 맞습니다.

```text
1. 시스템 핵심 그룹은 비활성화 불가
2. 일반 업무 코드는 비활성화 가능
3. 비활성화는 신규 선택만 막음
4. 기존 데이터 조회/라벨 표시는 유지
5. 드롭다운 조회와 라벨 조회 쿼리는 의도적으로 다르게 작성
```

이 방향이 가장 안전합니다.

그리고 실제 수정할 때 Mapper SQL은 프로젝트 규칙대로 `SELECT *` 없이 컬럼을 명시하고, 앞 콤마 스타일을 유지하는 게 맞습니다.
